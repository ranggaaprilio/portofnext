# Devtools UI Upgrade Plan

## Goal
Redesign the `apps/devtools` app shell so the sidebar and header feel materially different from the current product, while preserving the existing tool routing, categories, favorites, command palette, and theme support.

## Current Shell Findings
- `src/layouts/base.layout.vue` owns the current shell composition.
- The sidebar is a single `n-layout-sider` with a hero banner, collapsible category lists, and footer.
- The header is a thin utility row with menu toggle, home button, command palette trigger, and theme toggle.
- `src/components/MenuLayout.vue` provides a basic two-column `n-layout` shell with a mobile overlay.
- `src/components/CollapsibleToolMenu.vue` renders category sections with `n-menu` and persisted collapse state.
- `src/layouts/tool.layout.vue` adds a simple title, favorite action, and description above each tool.
- `src/stores/style.store.ts` only tracks dark mode, small-screen state, and whole-sidebar collapse.

## Selected Direction
Use a "developer workbench" shell instead of the current marketing-style gradient sidebar.

### Sidebar
Replace the current full-width category menu with a two-part navigation:
- A narrow left utility rail for brand, home, favorites, and high-level category switching.
- A secondary navigation panel showing the selected category's tools, recent/favorite tools, and optional quick filters.

This creates a clearly different structure from the current monolithic collapsible list and gives the UI a more app-like feel.

### Header
Replace the current icon row with a sticky workspace header:
- Left: current page context (`Home` or tool name), optional category label, and concise description.
- Center: prominent search / command launcher entry point.
- Right: quick actions such as favorite toggle, theme switch, and menu toggle on small screens.

For tool pages, the header should absorb most of what `tool.layout.vue` currently renders so the page title area becomes tighter and less repetitive.

## Implementation Plan

### Phase 1: Introduce a New App Shell
Create a shell that separates layout responsibilities more clearly.
- Refactor `src/components/MenuLayout.vue` into a more flexible shell or replace it with a new shell component.
- Support desktop split navigation and a mobile drawer/sheet pattern.
- Keep the current route/content slot contract so downstream tool pages stay stable.

Likely files:
- `src/components/MenuLayout.vue`
- `src/layouts/base.layout.vue`
- new shell/navigation components under `src/components/`

### Phase 2: Rebuild Sidebar Information Architecture
Shift from collapsible category stacks to rail + detail panel.
- Add a utility rail with brand, home, favorites, and category icons/labels.
- Convert tool navigation to a selected-category panel instead of rendering every category at once.
- Preserve favorite tools and category grouping from `src/tools/index.ts` and `src/tools/tools.store.ts`.
- Keep persisted UI state for selected category and panel visibility.

Likely files:
- `src/components/CollapsibleToolMenu.vue` or replacement components
- `src/components/MenuIconItem.vue`
- `src/tools/index.ts`
- `src/tools/tools.store.ts`
- `src/stores/style.store.ts`

### Phase 3: Redesign Header and Page Context
Make the header the primary control surface.
- Replace the current button strip in `base.layout.vue` with a structured sticky header.
- Promote the command palette trigger into a wider search field/button.
- Move theme toggle and route actions into a consistent action cluster.
- For tool pages, move title/description/favorite affordances into header-level context.

Likely files:
- `src/layouts/base.layout.vue`
- `src/layouts/tool.layout.vue`
- `src/components/NavbarButtons.vue`
- `src/modules/command-palette/command-palette.vue`

### Phase 4: Align Home and Tool Content Spacing
Adjust page bodies so the new shell feels intentional.
- Rework `Home.page.vue` top spacing and section rhythm to fit beneath the new header.
- Reduce duplicated hero-like spacing in `tool.layout.vue`.
- Ensure tool cards and tool content widths still work inside the new shell.

Likely files:
- `src/pages/Home.page.vue`
- `src/layouts/tool.layout.vue`
- `src/components/ToolCard.vue`

### Phase 5: Theme and Motion Pass
Update the visual language to support the new shell.
- Tune `src/themes.ts` for rail/panel/header surfaces and active states.
- Add any shared UnoCSS shortcuts needed for shell surfaces, borders, and sticky regions.
- Keep dark mode first-class and avoid low-contrast translucent treatments.

Likely files:
- `src/themes.ts`
- `src/ui/theme/themes.ts`
- `unocss.config.ts`

## Design Rules For The Upgrade
- Remove the current gradient hero treatment from the sidebar.
- Avoid a simple "same layout, new colors" outcome.
- Make navigation feel product-like rather than brochure-like.
- Preserve fast access to tools, favorites, and search.
- Keep desktop and mobile behavior clearly designed, not merely collapsed.
- Do not change tool internals unless required by shell layout integration.

## State And Interaction Changes
Recommended new UI state in `style.store.ts`:
- `activeSidebarSection` for current category or favorites view.
- `isNavigationOpen` for mobile drawer state.
- optional `isSidebarCompact` if desktop compact mode is desired.

Behavior to preserve or improve:
- Auto-close navigation on small screens after route change.
- Route-aware highlighting for active tool.
- Keyboard access to the command palette.
- Dark/light mode toggle.

## Validation Plan
- Run `pnpm --dir apps/devtools lint`
- Run `pnpm --dir apps/devtools typecheck`
- Run targeted UI smoke checks in the browser for desktop and mobile widths.
- If layout behavior changes significantly, add or update Playwright coverage for menu opening, route navigation, and command palette access.

## Risks
- `n-layout-sider` may constrain the desired split-rail experience; replacing it with a custom CSS grid/flex shell may be cleaner.
- Moving tool title/description into the header can create duplication if `tool.layout.vue` is not simplified at the same time.
- Mobile navigation can regress if overlay/drawer behavior is not route-aware.

## Recommendation
Preferred execution path:
1. Build a new shell with custom layout primitives instead of incrementally restyling the existing `n-layout-sider` setup.
2. Replace the sidebar with a rail + detail panel pattern.
3. Merge the current tool header into a sticky global header with route-aware actions.
4. Finish with theme polish and responsive cleanup.
